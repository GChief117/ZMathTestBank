Chapter 15 — A File System

15.1 A Programming Interface

I. Concept

  • Case study: Z specifies a simple file system — data structures + operations.
  • Programming interface: lists every operation and its intended effect.
  • File-level operations: read, write, add, delete — affect data within one file.
  • System-level operations: create, destroy, open, close — affect the system.
  • File management (create/destroy) vs file access (open/close).
  • add vs write: add extends a file with new data; write overwrites existing data.
  • Spec captures preconditions, postconditions, and error cases for each operation.

II. Vocab Table

TermPlain English

Programming interfaceList of operations on the file system with descriptions of their intended effects.

IV.1 Examples: Library Catalog, Hospital Records & Music Platform

A library catalog lets staff look up, update, shelve, and discard book entries.

  • Task: retrieve a book's details by call number.
  • Operation: read — returns the datum for an existing key.
  • Category: file-level — works inside one file.
  • Rationale: no new entry created, no data deleted — pure lookup.

A hospital records system lets doctors view, revise, admit, and discharge patient entries.

  • Task: change a patient's diagnosis on an existing chart.
  • Operation: write — overwrites existing data at that key.
  • Category: file-level — modifies data inside one file.
  • Rationale: key already exists; we replace its value, not extend the file.

A music platform lets users play, re-tag, upload, and remove tracks.

  • Task: upload a brand-new track that has no existing entry.
  • Operation: add — extends the file with a new key-data pair.
  • Category: file-level — inserts within a single file.
  • Rationale: key is fresh; add extends rather than overwrites.

IV.2 Examples: Office Complex, Airport Terminal & Data Centre

An office complex manager must provision and decommission tenant spaces.

  • Task: set up a brand-new tenant folder in the system.
  • Operation: create — adds a new file to the system.
  • Category: file management — changes which files exist.
  • Rationale: system-level: a new file appears, not just new data in an existing one.

An airport terminal controller opens and closes gates for boarding.

  • Task: make gate B7 available for passenger boarding.
  • Operation: open — makes a file available for reading and writing.
  • Category: file access — changes availability, not existence.
  • Rationale: gate already exists; we change its status, not create it.

A data centre admin retires a server that is no longer needed.

  • Task: permanently remove a decommissioned server's record.
  • Operation: destroy — removes a file from the system.
  • Category: file management — the file ceases to exist.
  • Rationale: not just deleting data inside a file; the file itself is removed.

IV.3 Examples: Restaurant Menu, Inventory Tracker & Student Roster

A restaurant menu system distinguishes adding a new dish from updating a price.

  • Task: a chef introduces an entirely new dish vs. adjusting an existing dish's price.
  • Operation: add for the new dish; write for the price change.
  • Category: both are file-level, but add extends while write overwrites.
  • Rationale: add requires the key to be fresh; write requires the key to already exist.

An inventory tracker distinguishes registering new stock from correcting a count.

  • Task: register a product that has never been stocked vs. fix a count error.
  • Operation: add for the new SKU; write for the correction.
  • Category: both file-level; the preconditions differ.

  • Operation: add for enrolment; write for the grade change.
  • Category: both file-level; precondition on key freshness separates them.
  • Rationale: confusing add with write leads to silent overwrites or duplicate-key errors.

15.2 Operations upon Files

I. Concept

  • Basic types: [Key, Data] — file content modelled abstractly.
  • File schema: contents : Key ⇻ Data — partial function, so no duplicate keys.
  • FileInit: contents′ = ∅ — a new file starts empty.
  • ΔFile: before-and-after state pair (File ; File′).
  • ΞFile: unchanged state — θFile = θFile′.
  • Read₀: ΞFile, k? ∈ dom contents, d! = contents k? — no side effect.
  • Write₀: ΔFile, contents′ = contents ⊕ {k? ↦ d?} — relational override.
  • Add₀: k? ∉ dom, contents′ = contents ∪ {k? ↦ d?}; Delete₀: contents′ = {k?} ⩤ contents.

II. Vocab Table

TermPlain English
Basic typeAn abstract, unstructured type introduced by declaration: [Key, Data].
Partial functionA function that maps each element of its domain to at most one element; not every possible input need be mapped.
Relational overrideOperator that updates a function by replacing or adding a maplet: f ⊕ {x ↦ y}.
Domain co-restrictionOperator that removes all maplets whose first element is in a given set: {k} ⩤ f.
FileInitSchema that sets the initial state: contents′ = ∅.
ΔFileSchema including both before-state and after-state: File and File′.
ΞFileSchema requiring the state to remain unchanged: θFile = θFile′.
Side effectA change to the state caused by an operation; read has none.

15.5 Formal Analysis

I. Concept

  • Formal analysis checks the spec for errors, contradictions, and hidden assumptions.
  • Initialisation theorem: ∃ SystemInit • true — proves the invariant is satisfiable.
  • Proof: file′ = ∅ is functional; open = ∅ ⊆ dom ∅ trivially.
  • Precondition calculation: pre Op reveals when each operation is defined.
  • Key results: e.g., pre Open₀ ⇔ n? ∈ dom file ∧ n? ∉ open.
  • Precondition table collects all results for review.
  • Raises design questions: e.g., can we destroy an open file? Formal analysis exposes such issues.
