Chapter 18 — Functional Refinement

18.1 Retrieve functions

I. Concept

  • A retrieve function f maps every concrete state to exactly one abstract state.
  • Cancellation law: R ⊆ S ⨞ f~ ⇔ R ⨞ f ⊆ S when f is total.
  • Simplified correctness: existentials over abstract states collapse to applications of f.
  • Table 18.1 rules: F-init-func-rel, F-corr-func-rel.
  • Weaker condition: ran R ⊆ dom f suffices; f need not be globally total.
  • Most practical refinements use retrieve functions — sorted list → set, array-prefix → sequence.
  • Functional retrieves compose by ordinary function composition.
  • Benefit: mechanised proofs are easier; the retrieve computes the abstract view.

The cancellation law states:

Correct. Post-compose with f instead of pre-composing with f~.

  • Retrieve: f total → cancellation applies.
  • Simplification: avoids reasoning about the inverse.

V.7 Practice: Cancellation Law Application

Correct. Move f to the other side of ⊆.

  • Simplification: no need to construct the inverse relation.

Correct. Concrete init then retrieve must land in abstract init.

V.9 Practice: F-corr-func-rel

Correct. Restrict to applicable states, do concrete op, retrieve, and check it refines the abstract op.

  • Abstract: ao = abstract operation.
  • Concrete: co = concrete operation.

  • Simplification: uses the cancellation law to remove f~ in practice.

V.12 Practice: Why Functional

Retrieve functions simplify proofs because:

Correct. ∃ a • R(c, a) ∧ … becomes …[f(c)/a].

  • Concrete: c is the input.
  • Retrieve: f computes a from c.
  • Simplification: no search for witness; direct calculation.

V.14 Practice: Weaker Condition

The cancellation law requires f to be total. The weaker condition that suffices is:

Correct. f need only be defined on states that R can reach.

  • Abstract: S stays as-is.

  • Retrieve: partial f suffices if ran R ⊆ dom f.
  • Simplification: relaxes the totality requirement.

18.2 Functional refinement

I. Concept

  • Functional refinement requires proving ∀ C • ∃1 A • R — the retrieve is a total function (F-func).
  • F-init-func: ∀ A'; C' • CI ∧ R' ⇒ AI.
  • F-corr-func: correctness + applicability — two obligations.
  • Correctness: ∀ A; A'; C; C' • pre AO ∧ R ∧ CO ∧ R' ⇒ AO.
  • Applicability: ∀ A; C • pre AO ∧ R ⇒ pre CO.
  • These rules are simpler than general data refinement — no existentials.
  • If the retrieve is defined by equations (e.g., s = ran l), it is automatically functional.
  • Most realistic specifications use this pattern.

II. Vocab Table

TermPlain English
Functional refinementData refinement whose retrieve is a total function; proof obligations reduce to equations.
F-func∀ C • ∃₁ A • R — every concrete state has exactly one abstract counterpart.
F-init-func∀ A'; C' • CI ∧ R' ⇒ AI — concrete init under retrieve implies abstract init.
F-corr-funcConcrete op + retrieve must imply abstract op; and abstract precondition + retrieve must imply concrete precondition.
Commuting diagramTwo paths around a square (concrete-then-retrieve vs retrieve-then-abstract) give the same result.

18.3 Calculating data refinements

I. Concept

  • Weakest refinement W: A ⊑ W and for all C refining A, W ⊑ C.
  • Calculating operations: wo == f ⨞ ao ⨞ f~ — retrieve, abstract-op, un-retrieve.
  • Proof: f~ ⨞ wo = ao ⨞ f~ by relational calculus (using surjectivity).
  • Calculation: derive concrete form step-by-step from abstract + retrieve — no guessing.
  • With schemas: CI ≅ CI ⨞ F'; CO ≅ F ⨞ AO ⨞ F' (Table 18.4).
  • Result is correct by construction — each step is a valid law application.
  • Produces both the concrete implementation and its proof simultaneously.
  • Requires f to be total and surjective (or dom ao ⊆ ran f weaker condition).

18.4 Refining promotion

I. Concept

  • Promotion lifts a local op on one instance to a global op on a collection.
  • Monotonicity: promotion is monotonic with respect to refinement.
  • Key result: “the refinement of a promotion is the promotion of the refinement.”
  • Simplest form: P ≅ [ f : I ⇸ S ] — a framing schema indexes instances.
  • Local refinement: refine data type S to T with retrieve r : T → S.
  • Pointwise retrieve: global retrieve g = f ⨞ r — compose the collection function with the local retrieve.
  • Benefit: prove refinement once for the local instance; the global refinement follows for free.
  • The framing schema is preserved verbatim; only the inner data type changes.

II. Vocab Table

TermPlain English
PromotionLifting a local operation on one instance to a global operation on a collection via a framing schema.
MonotonicIf a local data type refines, the promoted global op also refines — refinement of a promotion is the promotion of the refinement.
Framing schemaA schema selecting one instance from the collection and relating its change to the global state change.
Pointwise retrieveApply the local retrieve to each instance in the collection: g = f ⨞ r.
Local / globalLocal = one instance; global = the whole collection.

III. Symbol Table
