Chapter 12 — Schema Operators

12.1 Conjunction

I. Concept

  • Schema conjunction: S ∧ T merges declarations, conjoins predicates.
  • Type compatibility: shared variables must have matching types, else undefined.
  • Result: new schema introduces both sets of variables, imposes both constraints.
  • Separation of concerns: specify aspects independently, combine via ∧.
  • Schema inclusion: including a schema in a declaration part equals conjunction.
  • Abstract data type: variables + operations encapsulated in a schema.
  • State as schema: object of schema type represents a system state; state invariant must always hold.
  • Characteristic binding (θ): θS maps each component of S to its value.

V.1 Practice: Schema Conjunction Basics

What does schema conjunction S ∧ T produce?

Correct. Conjunction merges declarations and ands predicates.

  • Schemas: S has (a : A, b : B | P); T has (b : B, c : C | Q).
  • Merge: declarations union to a, b, c; shared b must match type.
  • Predicate: P ∧ Q.
  • Result: (a : A, b : B, c : C | P ∧ Q).

V.2 Practice: Type Compatibility

What happens when two schemas share a variable with different types and are conjoined?

Correct. Shared variables must agree in type for conjunction to be defined.

  • Schemas: S declares x : A; T declares x : B where A ≠ B.
  • Merge: impossible — x cannot be both A and B.
  • Predicate: no valid predicate can be formed.
  • Result: S ∧ T is undefined.

12.2 Decoration

I. Concept

  • State invariant: schema components + constraint that must always hold.
  • Decoration: adding prime (') to each component → after state; predicate becomes P[a'/a, b'/b].
  • Before state / after state: operation includes both State and State'.
  • Input (?) / Output (!): suffixes on component names; NOT decorations — part of the name.
  • Delta convention (Δ): ΔSchema == Schema ∧ Schema' — state may change.
  • Xi convention (Ξ): ΞSchema == ΔSchema with θSchema = θSchema' — state unchanged.
  • Characteristic binding (θ): θState = before binding; (θState)' = decorated binding mapping to primed values.
  • Initialisation: only State' — no before-state reference; sets starting values.

12.3 Disjunction

I. Concept

  • Schema disjunction: S ∨ T merges declarations and disjoins predicates.
  • Type compatibility: shared variables in S and T must have matching types.
  • Alternatives: describes alternative branches in system behaviour.
  • Constraint movement: if a constraint lives only in one declaration, move it to the predicate before combining.
  • Total operation: defined for all before-state / input combinations (success ∨ error).
  • Partial operation: defined only for some before-state / input combinations.
  • Error schema: captures the failure branch, combined via disjunction with the success branch.

12.4 Negation

I. Concept

  • Schema negation: ¬ S keeps the same components but negates the predicate.
  • Normalised schema: for normalised S with predicate P, ¬ S simply has predicate ¬ P.
  • Declaration constraint: non-normalised schemas embed constraints in declarations (e.g. a : ℕ restricts to naturals).
  • Type widening: negation widens types (e.g. ℕ → ℤ) and negates the entire conjunction including declaration constraints.
  • Non-normalised negation: ¬ S widens every declaration type and negates (decl-constraints ∧ predicate).
  • Normalise first: converting to a normalised schema before negating yields a clearer result.
  • Use case: State' ∧ (¬ Init) asserts the system is not in its initial state.

12.5 Quantification and hiding

I. Concept

  • Schema quantification: ∀ or ∃ over chosen components → removes them from signature, preserves constraint.
  • Syntax: Q dec • S — every variable in dec must already be a component of S with the same type.
  • Universal quantification: ∀ b : B • S → component b removed, predicate wrapped in ∀ b : B • P.
  • Existential quantification: ∃ b : B • S → component b removed, predicate wrapped in ∃ b : B • P.
  • Not omission: quantified constraint is preserved via quantifier — not the same as if the component had never appeared.
  • Hiding: existential quantification used to remove internal / implementation components from a schema’s visible signature.
  • Schema piping (>>): connects output! of one operation to input? of another — matching pairs hidden via ∃.
  • Spot (•): separates declaration from predicate in a quantified expression.
