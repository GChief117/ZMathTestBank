Chapter 17 — Data Refinement and Schemas

17.1 Relations and Schema Operations

I. Concept

  • Operation schema: defines how state after relates to state before.
  • Meaning: every operation schema is a relation upon states.
  • Partiality: if the precondition is unsatisfied, the operation is undefined.
  • Totalisation: extend with ⊥ so every input maps to some output.
  • I/O form: domain = State × Input; range = State × Output.
  • Split / merge: translate between schema world and relational world via ρs = split ⨟ (ρ ∥ id) ⨟ merge.
  • Retrieve schema: includes both A and C with a linking predicate.
  • Retrieve relation: shows how abstract data can be recovered from concrete.

17.2 Forwards Simulation

I. Concept

  • Forwards simulation checks refinement by matching each concrete step with an abstract one.
  • Derive relations from schemas: r = {R • θA ↦ θC}, ao = {AO • (θA, i?) ↦ (θA', o!)}, co = {CO • (θC, i?) ↦ (θC', o!)}.
  • Initialisation sets: ai = {AI • θA'} and ci = {CI • θC'} — after-states produced at startup.
  • F-init: ∀ C' • CI ⇒ ∃ A' • AI ∧ R' — every concrete init has an abstract match.
  • F-corr applicability: ∀ A; C • pre AO ∧ R ⇒ pre CO — concrete defined wherever abstract is.
  • F-corr correctness: ∀ A; C; C' • pre AO ∧ R ∧ CO ⇒ ∃ A' • AO ∧ R' — concrete results match some abstract result.
  • The retrieve relation R links abstract to concrete; R' links their after-states.
  • Each operation pair generates a proof obligation — three rules total (F-init + two parts of F-corr).

17.3 Backwards Simulation

I. Concept

  • Backwards simulation: needed when postponed nondeterminism blocks forwards proof.
  • B-init rule: ∀ A′; C′ • CI ∧ R′ ⇒ AI — every abstract equivalent of an initial concrete state must itself be initial.
  • B-corr applicability: ∀ C • (∀ A • R ⇒ pre AO) ⇒ pre CO.
  • B-corr correctness: ∀ C • (∀ A • R ⇒ pre AO) ⇒ (∀ A′; C′ • CO ∧ R′ ⇒ ∃ A • R ∧ AO).
  • Anticipation: abstract “anticipates” concrete — for any concrete result, an abstract before-state exists that matches.
  • Forwards-only failure: forwards simulation fails when the concrete system resolves a nondeterministic choice later than the abstract system does.
  • Atomicity change: abstract sees one atomic step; concrete splits it into sub-steps — backwards simulation handles the timing gap.
  • Both directions possible: some refinements admit both forwards and backwards simulations under different operation correspondences.
