Chapter 16 — Data Refinement

16.1 Abstract and Concrete State

I. Concept

  • Abstract state models what the system is: declared with high-level types like ℙ X, X ⇸ Y, seq X.
  • Concrete state models how it runs: arrays, pointers, linked cells, hash buckets, B-trees.
  • Two schemas: AState == [ av : AType | InvA ] vs CState == [ cv : CType | InvC ]. Same behavior, different structure.
  • Refinement preserves observable behavior — what inputs go in and what outputs come out — while freely changing internals.
  • The concrete state usually has a representation invariant (InvC) that encodes "this data structure is well-formed."
  • Multiple concrete states may correspond to the same abstract state (e.g., many trees → same set).
  • Refinement is unidirectional: concrete refines abstract, not the reverse.
  • Used in stepwise development: start abstract, refine repeatedly until implementation-ready.

16.2 Retrieve Relation

I. Concept

  • The retrieve relation Retrieve is a schema containing both the abstract and the concrete state, linked by a predicate.
  • Form: Retrieve == [ Abs; Conc | rel(av, cv) ]. The predicate rel encodes "this cv represents this av."
  • Often functional (deterministic): given a concrete state, the abstract it represents is uniquely determined. Call it a retrieve function.
  • Retrieve is typically many-to-one: different concrete states may map to the same abstract (e.g., different list orderings map to the same set).
  • Must be surjective onto reachable abstract states: every abstract state we care about has at least one concrete representative.
  • The retrieve relation encodes the concrete's representation invariant implicitly — only well-formed concrete states participate.
  • In refinement proofs: prove pre-state Retrieve plus concrete op implies some abstract op's post-state Retrieve'.
  • Canonical examples: ran cu = au (list → set), {i ↦ cu(i) | i ∈ dom cu} = af (array → map), nodes(bst) = as (tree → set).

16.3 Forwards Simulation

I. Concept

  • Forwards simulation runs the concrete operation forward, then finds a matching abstract move.
  • Given: Retrieve holds pre-op, concrete op COp executes.
  • Show: there exists an abstract post-state av' such that AOp holds and Retrieve' holds.
  • Two proof obligations together: applicability (concrete applies wherever abstract does) + correctness (concrete result retrieves to valid abstract result).
  • Formal predicate: ∀ av, cv, cv' • Retrieve ∧ pre AOp ∧ COp ⇒ ∃ av' • AOp ∧ Retrieve'.
  • Works cleanly when the concrete is at least as deterministic as the abstract — or when the concrete's choices can be matched by specific abstract choices.
  • The default and most common refinement-proof style: build outward from pre to post, matching concrete behavior with abstract behavior.
  • Fails when the concrete resolves non-determinism the abstract has — then backwards simulation is needed.

IV.1 Example: Forward Simulation of Add

SymbolPlain English
⁀Sequence concatenation.
∪Set union.
∃Existential.

A messaging platform's abstract AddUser (au' = au ∪ {u?}) is refined by concrete AppendUser (cu' = cu ⁀ ⟨ u? ⟩). Prove forwards simulation.

  • Given: Retrieve (au = ran cu), u? ∉ au, and cu' = cu ⁀ ⟨ u? ⟩.
  • Show: au' = au ∪ {u?} and au' = ran cu'.
  • Proof step: ran cu' = ran (cu ⁀ ⟨ u? ⟩) = ran cu ∪ {u?} = au ∪ {u?} = au'. ✓
  • Natural language: "appending to the list corresponds to adding to the set" — Retrieve preserved.

IV.2 Example: Forward Proof Shape

SymbolPlain English
⇒Implication.
∃Existential.
∧Conjunction.

The standard forward obligation is a single formula mentioning pre-state Retrieve, concrete op, abstract pre, and an existential over post-abstract state.

  • Shape: Retrieve ∧ pre AOp ∧ COp ⇒ ∃ av' • AOp ∧ Retrieve'.
  • Math: forward simulation rule.
  • Strategy: pick av' via the retrieve (often av' = retrieve(cv')) and verify AOp.
  • Natural language: "from pre-retrieve and concrete step, pick an abstract post that matches via retrieve."

16.4 Backwards Simulation

I. Concept

  • Backwards simulation proves refinement by starting from the post-state and working backward.
  • Given: post-state Retrieve' (links av' and cv') and concrete op COp fired.
  • Show: there exists pre-abstract state av such that AOp holds and Retrieve holds at the start.
  • Formal predicate: ∀ av', cv, cv' • Retrieve' ∧ COp ⇒ ∃ av • AOp ∧ Retrieve.
  • Needed when the concrete makes determinate choices that the abstract left open (e.g., abstract "pick any element," concrete "pick head").
  • Forwards cannot handle such cases because choosing av' up-front commits you to a specific future which concrete's choice may contradict; backwards lets you pick av after seeing the concrete's move.
  • Together with forwards, they form a jointly complete proof system — every valid refinement is provable by one or both.
  • In practice: try forwards first; fall back to backwards when forwards stalls on non-determinism resolution.

IV.1 Example: Abstract "Pick Any" vs Concrete "Pick Head"

SymbolPlain English
headFirst element of a sequence.
∃Existential.
tailAll but the first.

A queue abstractly removes "any" item (x! ∈ ab; ab' = ab \ {x!}); concretely, it removes the head (x! = head cq; cq' = tail cq). Forwards can't match "any" with "head"; backwards succeeds.

  • Given post: ab' = ran cq' (retrieve') and concrete pop occurred (x! = head cq).
  • Find pre-abstract: av = ran cq; verify av = av' ∪ {x!} ⇒ AOp holds (picks x!).
  • Math: backwards simulation.
  • Natural language: "knowing what the concrete did, commit to an abstract history consistent with it."

IV.2 Example: Backward Proof Shape

SymbolPlain English
⇒Implication.
∃Existential.

The standard backward obligation existentially quantifies over the pre-abstract state and requires post-state Retrieve' to determine a matching abstract history.

  • Shape: Retrieve' ∧ COp ⇒ ∃ av • AOp ∧ Retrieve.
  • Math: backward simulation rule.
  • Strategy: pick av = retrieve(cv); verify both the abstract op and the pre-retrieve.
  • Natural language: "the concrete step rewrites to a consistent abstract history."

16.5 Applicability

I. Concept

  • The applicability condition guarantees the refinement introduces no new rejections.
  • Formal obligation: ∀ av, cv • pre AOp ∧ Retrieve ⇒ pre COp.
  • Intuition: "if the abstract spec says the op is defined here, the concrete implementation must also define it here."
  • If applicability fails, the concrete spec weakens the service: clients expect an op to succeed, but the concrete can refuse.
  • Applicability allows concrete precondition weakening — concrete may apply on more states than abstract requires (but never on fewer).
  • Proved by expanding pre AOp and Retrieve, simplifying, and showing pre COp holds.
  • When both operations are total (pre = true), applicability is trivial — always satisfied.
  • One of two obligations in forwards simulation; the other is correctness.

SymbolPlain English
prePrecondition.

An abstract dictionary's Add requires #as < MAX. The concrete BST-Insert must apply whenever that holds.

  • Given: pre Add ≡ #as < MAX; Retrieve ≡ as = nodes(bst).

IV.2 Example: Applicability Failure

SymbolPlain English
prePrecondition.
¬Rightarrow;Does not imply.

A concrete hash-Insert that requires a free slot in the backing array fails applicability if the abstract Add has no capacity constraint.

  • Abstract pre: true (Add always defined).
  • Concrete pre: ∃ i : dom arr • arr(i) = null (need free slot).
  • Obligation: true ∧ Retrieve ⇒ ∃ i • arr(i) = null. False when array is full; applicability fails.
  • Natural language: "abstract allows adding always, but concrete refuses when full" — refinement invalid as given; must either re-do the spec or add resize to the op.

16.6 Correctness

I. Concept

  • The correctness condition ensures the concrete operation's outcome is valid at the abstract level.
  • Formal obligation: ∀ av, cv, cv' • pre AOp ∧ Retrieve ∧ COp ⇒ ∃ av' • AOp ∧ Retrieve'.
  • Intuition: "after the concrete op runs, there's an abstract post-state it corresponds to, and the abstract op legally produces that state."
  • If correctness fails, the concrete produces an outcome the abstract wouldn't accept — a refinement bug.
  • Proved by: (1) pick av' = retrieve(cv'); (2) verify Retrieve' holds (often by calculation); (3) verify AOp relates av to av'.
  • Concrete may be more deterministic than abstract: the concrete's single outcome must correspond to some legal abstract outcome.
  • Allows post-condition strengthening: concrete may produce a more specific result than abstract allowed, but must still fall within the abstract's options.
  • Second of two forwards-simulation obligations; together with applicability, they are jointly sufficient for correct refinement.

IV.1 Example: Correctness of BST-Insert

SymbolPlain English
∪Set union.
∃Existential.
⇒Implication.

A BST-Insert refining a set-Add is correct when nodes(bst') = nodes(bst) ∪ {x?} — the post-tree's elements exactly match the post-set.

  • Obligation: Retrieve ∧ pre Add ∧ Insert ⇒ ∃ as' • Add ∧ Retrieve'.
  • Pick: as' = nodes(bst') (from Retrieve').
  • Verify AOp: as' = as ∪ {x?} follows from nodes(bst') = nodes(bst) ∪ {x?} and as = nodes(bst). ✓
  • Natural language: "inserting into the BST adds exactly one element to the represented set."

IV.2 Example: Correctness Failure — Buggy Delete

SymbolPlain English
\Set difference.
¬Rightarrow;Does not imply.

A concrete BST-Delete that removes the wrong node (e.g., always removes the root) fails correctness against an abstract Remove(x?).

  • Abstract: as' = as \ {x?}.
  • Buggy concrete: bst' = removeRoot(bst) (ignores x?).
  • Failure: nodes(bst') = nodes(bst) \ {root(bst)} ≠ as \ {x?} when root(bst) ≠ x?.
  • Natural language: "concrete removed wrong element; post-state doesn't retrieve to intended abstract post." Correctness obligation unprovable.
