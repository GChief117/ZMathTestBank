Chapter 19 — Refinement Calculus

19.1 Specification Statement

I. Concept

  • A specification statement has the form w : [ pre, post ] — frame w, precondition pre, postcondition post.
  • The frame lists variables that may change; everything else stays fixed.
  • Pre: assumption on the initial state. Post: guarantee about the final state.
  • Specification statements are the starting point for refinement calculus — they are refined stepwise into code.
  • An op with trivially-satisfiable post is skip; an infeasible post gives magic.
  • Specification statements unify Z schemas and imperative-code specifications.
  • Pre/post uses primed variables (x') to reference the final state.
  • Calculus: rewrite w : [ pre, post ] into simpler statements until code remains.

19.2 Assignment

I. Concept

  • Assignment x := e updates one variable.
  • Assignment rule: w, x : [ pre, post ] ⊆ x := e when pre ⇒ post[e/x'] (post is made true by setting x to e).
  • Weakening pre and strengthening post are valid refinement steps.
  • A spec is refined to an assignment when the postcondition is a simple equation x' = e.
  • The frame must include x; non-frame variables are untouched.
  • Assignments are the atomic code steps of refinement calculus.
  • Feasibility is preserved: a feasible spec refined to assignment gives executable code.
  • Think of assignment as the final step of a derivation, converting specification to code.

19.3 Logical Constants

I. Concept

  • A logical constant is declared with con c : T | P(c) • spec — names an unknown fixed value with property P.
  • Logical constants let us record the pre-state value during a derivation, referring to it in the post.
  • Introduction rule: if c doesn't appear in spec, adding a con declaration is a valid step.
  • Elimination: a con is removed when its role is done — it never appears in final executable code.
  • Logical constants are auxiliary: they exist only in the proof/derivation.
  • A typical pattern: con c • v : [ v = c, v' = f(c) ] records the initial value of v as c.
  • Scoping uses begin / end blocks: the constant is valid inside the block only.
  • Logical constants are the refinement-calculus analog of "ghost variables" in Hoare logic.

19.4 Sequential Composition

I. Concept

  • Sequential composition splits [ pre, post ] into two parts executed one after the other.
  • Rule: w : [ pre, post ] ⊆ w : [ pre, mid ] ; w : [ mid, post ] — pick a mid predicate true after the first half.
  • The trick is choosing mid carefully so both sub-specs are easier to refine.
  • Logical constants often appear: con c records the initial value so the mid can reference it.
  • Sequential composition introduces the ; operator between specs / code.
  • Each half is refined independently (monotonicity); compose the results.
  • The art is picking a mid that splits the work evenly — too weak or too strong hurts.
  • This rule is how most algorithms are constructed: do step A, then step B.

19.5 Conditional Statements

I. Concept

  • Conditional if G then S1 else S2 fi chooses a branch based on guard G.
  • Refinement rule: w : [ pre, post ] ⊆ if G then w : [ pre ∧ G, post ] else w : [ pre ∧ ¬G, post ] fi.
  • Each branch is refined assuming its guard's truth value.
  • A multi-way if ... elif ... elif ... fi is valid when the guards exhaust the pre-state.
  • Non-exhaustive branches require adding else to avoid undefined behavior.
  • Conditional introduces case analysis — powerful when the postcondition splits by cases.
  • Each branch is a separate refinement; the whole spec is the merge.
  • The guard must be computable from the current state.

IV.1 Example: Max of Two via Conditional

SymbolPlain English

A thermostat's max function picks the larger reading via a guard a ≥ b.

  • Spec: m : [ true, m' = max(a, b) ].
  • Guard: a ≥ b.

19.6 Iteration

I. Concept

  • Iteration while G do S od repeats body S while guard G holds.
  • Refinement requires a loop invariant I: true before, preserved by body, and I ∧ ¬G ⇒ post.
  • A variant (or termination function) V proves termination: strictly decreases each iteration, bounded below.
  • Rule: w : [ I, I ∧ ¬G ] ⊆ while G do w : [ I ∧ G, I ∧ V < V0 ] od.
  • Loop body must preserve invariant + strictly reduce variant.
  • Choosing a good invariant is the key engineering step — it captures "what's true throughout."
  • Variants prevent infinite loops by forcing progress.
  • Iteration + conditional + sequential composition = all structured imperative programs.
