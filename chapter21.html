Chapter 21 — An Operating System Scheduler

21.1 Processes

I. Concept

  • Scheduler: decides which process uses the single processor and when.
  • Single processor: shared among many processes; only one runs at a time.
  • Current: the process running on the CPU — at most one at any instant.
  • Ready: waiting for the processor, eligible to be dispatched next.
  • Blocked: waiting on another resource or event, not the processor.
  • Lifecycle: processes are created before scheduling, destroyed after use.
  • Control returns to the kernel on a service call or interrupt; scheduler then dispatches next.
  • Rigorous development: uses auxiliary data type and refinement for implementation.

V.1 Practice: Process State Definitions

Which of the following best describes the three states a process can occupy?

Correct. The textbook defines exactly three states: current, ready, and blocked.

  • State: Current = on the processor; Ready = waiting for processor; Blocked = waiting on another resource.
  • Transition: Processes move between these three states via scheduler operations.
  • Scheduler: It decides which ready process becomes current.
  • Effect: At most one current process at any time.

V.2 Practice: Scheduler Role

What is the primary role of the scheduler?

Correct. The scheduler controls access to the processor among competing processes.

  • State: Multiple processes exist, each in current, ready, or blocked.
  • Transition: When current yields, control returns to the kernel.
  • Scheduler: It picks the next ready process to become current.
  • Effect: Only one process runs at a time on the single processor.

V.24 Practice: Service Call vs Interrupt

At a call centre, Caller A hangs up voluntarily. At a taxi stand, a traffic officer orders Taxi A to move.

Which correctly classifies these two events?

Correct. A service call is voluntary; an interrupt is an external forced signal.

V.25 Practice: State Exclusivity

A student argues that a plane at an airport could be simultaneously ready and blocked — taxiing but also waiting for a weather update.

Is this possible under the scheduler model?

Correct. The three states are mutually exclusive; each process occupies exactly one.

  • State: Current, ready, and blocked form a partition of active process states.
  • Transition: A process moves from one state to another, never occupying two.
  • Scheduler: It only considers ready processes for dispatch.
  • Effect: If a plane needs a weather update, it is blocked (not ready) until the update arrives.

21.2 Specification

I. Concept

  • Process count: the system manages up to n : ℕ processes.
  • Process identifier (PId) == 1..n; nullPId == 0 marks “no process.”
  • OptPId == PId ∪ {nullPId} — current may hold a real pid or null.
  • AScheduler state: current : OptPId, ready : ℙ PId, blocked : ℙ PId, free : ℙ PId.
  • Partition invariant: ⟨{current} \ {nullPId}, ready, blocked, free⟩ partition PId.
  • Operations: Dispatch, Time-out, Block, Wake-up, Create, Destroy.
  • Initialisation: current′ = nullPId, ready′ = ∅, blocked′ = ∅, free′ = PId.
  • Destroy ≅ ADestroyCurrent ∨ ADestroyReady ∨ ADestroyBlocked.

21.3 Chains

I. Concept

  • A chain is a finite injection links : PId ⤔ PId plus start, end : OptPId.
  • Start: in dom links but not ran links; End: in ran links but not dom links.
  • Empty chain: links = ∅ ⇒ start = nullPId ∧ end = nullPId.
  • Reachability: every element reachable from start via links+ (transitive closure).
  • Push: PushEmpty (end = nullPId) ∨ PushNonEmpty (links' = links ∪ {end ↦ p?}).
  • Pop: PopSingleton (links = ∅) ∨ PopMultiple (start' = links(start), links' = {start} ⩤ links).
  • Delete: DeleteStart (= Pop) ∨ DeleteEnd (links' = links ⊳ {end}) ∨ DeleteMiddle (links' = {p?} ⩤ links ⊕ {links~ p? ↦ links p?}).
  • Key tools: domain anti-restriction (⩤), range restriction (⊳), override (⊕).

21.4 Design

I. Concept

  • The concrete scheduler uses three schema renamings of Chain: ReadyChain, BlockedChain, FreeChain.
  • Renaming syntax: ReadyChain ≅ Chain[rstart/start, rend/end, rlinks/links, rset/set].
  • CScheduler includes the three chains, current : OptPId, and chainstore : PId → OptPId.
  • Partition invariant: ({current} \ {nullPId}), rset, bset, fset partition PId.
  • Link recovery: rlinks = rset ⩤ chainstore ⊳ rset; similarly for blinks, flinks — one shared map, three views.
  • Current constraint: current ≠ nullPId ⇒ chainstore(current) = nullPId.
  • Init: ReadyChainInit, BlockedChainInit (empty), FreeChainFull (fset' = PId), current' = nullPId.
  • Operations: CDispatch, CTimeOut, CBlock, CWakeUp, CCreate, CDestroy — each built from renamed Push/Pop/Delete on the relevant chain, constituting data refinement of the abstract scheduler.

21.5 Correctness

I. Concept

  • A retrieve relation connects the concrete state to the abstract state: ready = rset, blocked = bset, free = fset.
  • This is a functional refinement — the retrieve relation is functional from concrete to abstract.
  • Total function proof: ∀ CScheduler • ∃1 AScheduler • RetrScheduler.
  • Initialisation correctness: CSchedulerInit ∧ Retr′ ⊢ ASchedulerInit.
  • Correctness condition: pre AOp ∧ Retr ∧ COp ∧ Retr′ ⊢ AOp.
  • The applicability condition is automatically satisfied for functional refinements.
  • This pattern must be checked for every pair of abstract and concrete operations: Dispatch, TimeOut, Block, WakeUp, Create, Destroy.
  • Successful verification confirms the chain-based design faithfully implements the set-based specification.

Initialisation correctnessProof that the concrete initial state maps via Retr to the abstract initial state.
Applicability conditionRequirement that the concrete operation is defined whenever the abstract precondition holds.
Correctness conditionProof that the concrete operation preserves preconditions and simulates the abstract operation.
Concrete stateImplementation-level state using chains (CScheduler).
Abstract stateSpecification-level state using sets (AScheduler).
Functional refinementConcrete-to-abstract refinement where the retrieve relation is a total function.

V.1 Practice: Retrieve Relation Purpose

What is the primary purpose of a retrieve relation in data refinement?

Correct. The retrieve relation connects implementation to specification.

  • Retrieve: Maps concrete components (chains) to abstract components (sets).
  • Applicability: Without it, there is no way to compare concrete and abstract.
  • Correctness: Every proof obligation references the retrieve relation.
  • Conclusion: It is the bridge between design and specification.

V.2 Practice: Correctness Proof Steps

Which three proof obligations must hold for a correct functional refinement?

Correct. These are the three standard data-refinement obligations.

  • Retrieve: Must be a total function from concrete to abstract.
  • Applicability: Initialisation must map correctly.
  • Correctness: Each operation pair must satisfy the correctness condition.
  • Conclusion: All three together guarantee faithful implementation.
